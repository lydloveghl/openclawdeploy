# OpenClaw 一键部署与使用手册

这份手册是给 `openclawdeploy` 配套用的。

目标很简单：你把 OpenClaw 装起来，然后知道它怎么接模型、怎么接渠道、怎么配 skills、怎么查状态、怎么排错。

---

## 1. 这套部署方案的思路

`openclawdeploy` 并不是重写一套 OpenClaw 安装器，而是：

1. **优先复用 OpenClaw 官方安装脚本**
2. 在它上面补一层 **可交互安装向导**
3. 再补上自动化动作：
   - 写 `~/.openclaw/openclaw.json`
   - 配置常见模型认证
   - 支持跳过模型接入
   - 支持自定义大模型 provider
   - 安装 Feishu 插件
   - 安装并启动 Gateway
   - 配置 skills

这样做的好处是：

- 更贴近官方支持路径
- 后续升级不容易打架
- 出问题时更容易按官方文档排查

## 2. 平台支持说明

### macOS

推荐，支持较完整。

安装：

```bash
cd ~/openclawdeploy
chmod +x install.sh scripts/deploy.mjs
./install.sh
```

### Linux

推荐，Gateway 支持成熟。

安装：

```bash
cd ~/openclawdeploy
chmod +x install.sh scripts/deploy.mjs
./install.sh
```

### Windows

**官方推荐优先走 WSL2**，因为 OpenClaw 在 Linux 运行时更稳。

但本项目也提供原生 PowerShell 入口：

```powershell
cd $HOME\openclawdeploy
powershell -ExecutionPolicy Bypass -File .\install.ps1
```

如果你是正式生产环境，我建议：

- Windows 本机开发测试可以原生跑
- 真正常驻服务优先：**WSL2 / Linux / macOS**

## 3. 图形安装器怎么用

如果你不想通过终端部署，现在可以直接双击启动：

### macOS

- 双击 `OpenClawDeploy.command`
- 或先生成 `.app`：

```bash
node ./scripts/package-macos-app.mjs
open ./dist/OpenClawDeploy.app
```

### Linux

- 双击 `OpenClawDeploy.sh`

### Windows

- 双击 `OpenClawDeploy.bat`

启动后会自动打开浏览器图形安装器。你可以直接在网页里填写：

- 模型配置
- 自定义 provider
- Telegram / Feishu
- Skills
- dry-run / 正式执行

安装过程和日志也会直接显示在页面里。

## 4. 交互式安装向导怎么用

如果你还是想走终端，但不想记命令参数，现在最简单的用法就是：

```bash
./install.sh
```

或者：

```bash
./install.sh --interactive
```

向导会依次问你：

1. 工作目录
2. 要不要现在接入大模型
3. 接哪个提供方：
   - OpenRouter
   - OpenAI
   - Anthropic
   - 自定义 provider
   - 先跳过
4. 要不要接 Telegram
5. 要不要接 Feishu / Lark
6. 要不要配置 skills
7. 要不要执行 doctor / gateway install / gateway start
8. 要不要先 dry-run
9. 如果已有配置，要不要覆盖并自动备份

### 向导的核心特点

- **允许跳过大模型接入**
- **允许 API Key 先留空**，后面再手动补
- **支持自定义 provider**
- **支持 skills 配置**
- **支持最后确认执行计划**

## 5. 如果你不想交互，也可以命令行直传参数

### 4.1 跳过模型接入

```bash
./install.sh --provider none
```

这会：

- 安装 OpenClaw
- 写基础配置
- 不配置默认模型
- 不执行模型认证

非常适合先把环境搭起来，后面再慢慢接模型。

### 4.2 OpenRouter

```bash
./install.sh --provider openrouter --api-key <OPENROUTER_API_KEY>
```

默认模型：

```text
openrouter/anthropic/claude-sonnet-4-5
```

### 4.3 OpenAI

```bash
./install.sh --provider openai --api-key <OPENAI_API_KEY>
```

默认模型：

```text
openai/gpt-5.4
```

### 4.4 Anthropic

```bash
./install.sh --provider anthropic --api-key <ANTHROPIC_API_KEY>
```

默认模型：

```text
anthropic/claude-opus-4-6
```

### 4.5 自定义 provider

如果你有自己的代理、企业网关，或者兼容 OpenAI / Anthropic 协议的第三方模型服务，可以这样接：

```bash
./install.sh \
  --provider custom \
  --custom-provider-id myproxy \
  --custom-api openai-completions \
  --custom-base-url https://example.com/v1 \
  --custom-model-id gpt-4.1 \
  --custom-model-name "My Proxy GPT-4.1" \
  --custom-api-key sk-xxx
```

### 支持的自定义 adapter

- `openai-completions`
- `openai-responses`
- `anthropic-messages`
- `google-generative-ai`

### 写入后的大致配置结构

```json5
{
  agents: {
    defaults: {
      workspace: "~/.openclaw/workspace",
      model: { primary: "myproxy/gpt-4.1" },
    },
  },
  models: {
    mode: "merge",
    providers: {
      myproxy: {
        baseUrl: "https://example.com/v1",
        api: "openai-completions",
        apiKey: "sk-xxx",
        models: [{ id: "gpt-4.1", name: "My Proxy GPT-4.1" }],
      },
    },
  },
}
```

## 6. 配置文件在哪

OpenClaw 默认读取：

```text
~/.openclaw/openclaw.json
```

这是 **JSON5**，所以允许：

- 注释
- 尾逗号

如果脚本发现已有配置：

- 交互模式下会先问你要不要覆盖
- 非交互模式下需要你显式加 `--force`

覆盖前会自动备份，例如：

```text
~/.openclaw/openclaw.json.bak.2026-03-09T12-00-00.000Z
```

## 8. 接入大模型 API

这套项目内置了三种最常见的认证快捷方式：

- OpenAI
- Anthropic
- OpenRouter

如果你不用这三种，也可以：

- `--provider none`，先装好 OpenClaw
- 或者 `--provider custom`，写入自定义 provider
- 或者安装完成后再手动 `openclaw onboard`

### 7.1 手动走官方 onboard

如果你不想在命令里传 key，也可以安装完成后自己执行：

```bash
openclaw onboard
```

或者按官方推荐方式：

#### OpenAI

```bash
openclaw onboard --openai-api-key "$OPENAI_API_KEY"
```

#### Anthropic

```bash
openclaw onboard --anthropic-api-key "$ANTHROPIC_API_KEY"
```

#### OpenRouter

```bash
openclaw onboard --auth-choice apiKey --token-provider openrouter --token "$OPENROUTER_API_KEY"
```

## 9. 接入 Telegram

OpenClaw 当前对 Telegram 是原生支持的。

### 8.1 先创建 Bot

1. 打开 Telegram
2. 找 **@BotFather**
3. 执行 `/newbot`
4. 保存 Bot Token

### 8.2 部署时直接带上参数

```bash
./install.sh \
  --provider openrouter \
  --api-key <OPENROUTER_API_KEY> \
  --with-telegram \
  --telegram-bot-token <TELEGRAM_BOT_TOKEN>
```

### 8.3 生成的核心配置

```json5
{
  channels: {
    telegram: {
      enabled: true,
      botToken: "123:abc",
      dmPolicy: "pairing",
      groups: {
        "*": { requireMention: true },
      },
    },
  },
}
```

### 8.4 启动后怎么让私聊生效

```bash
openclaw gateway start
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

### 8.5 如果你要让群里默认不需要 @

部署时加：

```bash
--telegram-require-mention false
```

或者手动改配置：

```json5
{
  channels: {
    telegram: {
      groups: {
        "*": { requireMention: false },
      },
    },
  },
}
```

## 10. 接入 Feishu / Lark

OpenClaw 对 Feishu 的支持来自 **单独插件**，不是主程序内置。

### 9.1 插件安装

本项目在你启用 `--with-feishu` 时，会自动执行：

```bash
openclaw plugins install @openclaw/feishu
```

### 9.2 去飞书开放平台创建应用

你需要：

1. 打开 Feishu Open Platform
2. 创建企业自建应用
3. 打开 **Bot** 能力
4. 记录：
   - App ID
   - App Secret
5. 事件订阅选择 **长连接（WebSocket）**
6. 开启 `im.message.receive_v1`
7. 发布应用

### 9.3 部署时带参数

```bash
./install.sh \
  --provider openrouter \
  --api-key <OPENROUTER_API_KEY> \
  --with-feishu \
  --feishu-app-id cli_xxx \
  --feishu-app-secret yyy \
  --feishu-bot-name 公司助手
```

如果是国际版 Lark：

```bash
./install.sh \
  --provider openrouter \
  --api-key <OPENROUTER_API_KEY> \
  --with-feishu \
  --feishu-domain lark \
  --feishu-app-id cli_xxx \
  --feishu-app-secret yyy
```

### 9.4 生成的核心配置

```json5
{
  channels: {
    feishu: {
      enabled: true,
      domain: "feishu",
      dmPolicy: "pairing",
      accounts: {
        default: {
          appId: "cli_xxx",
          appSecret: "xxx",
          botName: "OpenClaw",
        },
      },
    },
  },
}
```

### 9.5 启动后检查

```bash
openclaw gateway status
openclaw logs --follow
```

## 11. 配置 Skills

现在 `openclawdeploy` 已经支持把 skills 配置直接写进 `~/.openclaw/openclaw.json`。

### 10.1 Skills 配置在哪

都在：

```json5
{
  skills: {
    ...
  },
}
```

### 10.2 这版支持写哪些字段

- `skills.allowBundled`
- `skills.load.extraDirs`
- `skills.load.watch`
- `skills.load.watchDebounceMs`
- `skills.install.preferBrew`
- `skills.install.nodeManager`
- `skills.entries.<skillKey>.enabled`
- `skills.entries.<skillKey>.apiKey`
- `skills.entries.<skillKey>.env`
- `skills.entries.<skillKey>.config`

### 10.3 命令行示例

```bash
./install.sh \
  --configure-skills \
  --skills-extra-dirs ~/my-skills,~/team-skills \
  --skills-allow-bundled weather,apple-notes \
  --skills-node-manager pnpm \
  --skills-prefer-brew true \
  --skill-entry-json '{"key":"sag","enabled":false}' \
  --skill-entry-json '{"key":"nano-banana-pro","apiKey":"xxx","env":{"GEMINI_API_KEY":"xxx"}}'
```

### 10.4 生成的配置示例

```json5
{
  skills: {
    allowBundled: ["weather", "apple-notes"],
    load: {
      extraDirs: ["~/my-skills", "~/team-skills"],
      watch: true,
      watchDebounceMs: 250,
    },
    install: {
      preferBrew: true,
      nodeManager: "pnpm",
    },
    entries: {
      sag: { enabled: false },
      "nano-banana-pro": {
        apiKey: "xxx",
        env: {
          GEMINI_API_KEY: "xxx",
        },
      },
    },
  },
}
```

### 10.5 Skills 配置什么时候生效

一般来说：

- **新会话**最稳妥
- watcher 开着时，下一次 agent turn 也可能自动刷新

如果你改完感觉没生效，先：

```bash
openclaw gateway restart
```

然后新开一个会话再试。

## 12. 接入 QQ

### 11.1 先说实话

根据当前本机 OpenClaw 官方文档，**没有原生 QQ channel 文档，也没有内置 QQ 官方接入说明**。

所以这套项目不会骗你说：

> “QQ 已经像 Telegram 一样开箱即用。”

不是。当前更现实的做法是：

- **NapCat / OneBot / go-cqhttp 一类 QQ 适配层**
- 再做一个 bridge，把 QQ 消息转成 OpenClaw 可以消费的输入
- 把 OpenClaw 输出再回写到 QQ

详细说明看：

- [QQ接入说明.md](./QQ接入说明.md)

## 13. 典型部署方案推荐

### 个人使用，最省事

- 平台：macOS / Linux
- 模型：OpenRouter
- 渠道：Telegram 或 Feishu
- skills：按需配，不要一次开太多

### 团队内部使用

- 平台：Linux 服务器
- 模型：OpenAI / Anthropic / OpenRouter / 企业自定义代理
- 渠道：Feishu
- skills：统一放共享目录，再通过 `skills.load.extraDirs` 加载

### Windows 用户

- 测试：原生 PowerShell 可以先跑
- 长期稳定：建议 WSL2 或单独 Linux 主机

## 14. 故障排查顺序

如果部署完成后不好使，我建议按这个顺序查：

### 第一步：CLI 在不在

```bash
openclaw --version
```

### 第二步：总状态

```bash
openclaw status
```

### 第三步：Gateway 状态

```bash
openclaw gateway status
```

### 第四步：重启服务

```bash
openclaw gateway restart
```

### 第五步：看日志

```bash
openclaw logs --follow
```

### 第六步：doctor

```bash
openclaw doctor --non-interactive
```

### 第七步：检查配置文件

```bash
cat ~/.openclaw/openclaw.json
```

重点看：

- model 是否写对
- custom provider 的 baseUrl / api / model id 是否写对
- token / appId / appSecret 是否为空
- telegram / feishu 是否 enabled
- skills 配置有没有写错结构
- 是否不小心覆盖掉了旧配置

## 15. 后续升级建议

如果你准备继续把这套东西做得更像真正的“生产部署工具”，建议优先补这几件：

1. `.env` 文件支持
2. 读取并合并现有 `openclaw.json`，而不是整文件覆盖
3. Docker Compose 版本
4. Linux systemd / macOS launchd 的更细粒度检查
5. QQ OneBot bridge 落地
6. 渠道和模型的健康检查

---

## 16. 最后一句

如果你只是想最快跑起来：

- 模型先用 **OpenRouter**
- 渠道先接 **Telegram 或 Feishu**
- skills 先少量配置
- QQ 先别强行一步到位

这样成功率最高，也最接近 OpenClaw 官方支持路径。
